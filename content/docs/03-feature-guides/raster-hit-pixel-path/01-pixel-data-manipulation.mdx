# Pixel data — read, write, manipulate {#01-pixel-data-manipulation}

Back: [Raster overview](/docs#00-advanced-raster-overview)

All pixel IO goes through **`painter.pixels`**, implemented by **`PixelDataCreator`** (**`lib-next/pixels/pixel-data-creator.ts`**). The incoming argument is always a **PNG** **`Buffer`** or a **`CanvasResults`** object whose **`.buffer`** is decoded; work happens on an internal **`@napi-rs/canvas`** surface via **`getImageData`** / **`putImageData`**.

---

## Why use this instead of Sharp-only helpers?

- You need **exact `ImageData` layout** (RGBA, row-major) for algorithms, game maps, or custom shaders ported to JS.
- You want a **small region** or **per-pixel** updates without pulling in heavier pipelines.
- **`painter.pixels.manipulate`** exposes a simple **`processor`** callback or a fixed set of **CPU filters** for quick experiments.

For whole-image convolution, resize-at-scale, or format conversion, **`painter.image`** (Sharp-backed) is usually faster — combine both: sample with **`pixels`**, bulk work with **`image`**.

---

## API surface (`PainterPixels`)

| Method | Returns |
|--------|---------|
| **`getData(buffer, options?)`** | **`Promise<PixelData>`** — optional crop **`{ x, y, width, height }`** |
| **`setData(buffer, pixelData, options?)`** | **`Promise<Buffer>`** (PNG) — paste **`ImageData`**-shaped bytes at **`x,y`**, optional dirty sub-rect |
| **`manipulate(buffer, options)`** | **`Promise<Buffer>`** — **`processor`** or **`filter`** + optional **`region`** |
| **`getColor(buffer, x, y)`** | **`Promise<{ r, g, b, a }>`** |
| **`setColor(buffer, x, y, color)`** | **`Promise<Buffer>`** |

---

## `PixelData`

| Field | Meaning |
|-------|---------|
| **`data`** | **`Uint8ClampedArray`**, row-major RGBA (**`4 × width × height`** bytes) |
| **`width`**, **`height`** | Region size in pixels |
| **`colorSpace`** | Set to **`'srgb'`** on read |

Index for pixel **`(x, y)`** inside the region: **`idx = (y * width + x) * 4`**, then **`r,g,b,a`** at **`idx`…`idx+3`**.

---

## `getData(canvasBuffer, options?)`

**`options`**: **`x`**, **`y`**, **`width`**, **`height`** — crop (defaults: full image). Throws if the rectangle exceeds image bounds.

```typescript
const region = await painter.pixels.getData(buf, { x: 10, y: 10, width: 64, height: 64 });
// region.data.length === 64 * 64 * 4
```

---

## `setData(canvasBuffer, pixelData, options?)`

Copies **`canvasBuffer`** to a working canvas, then **`putImageData`** for **`pixelData`**.

| Option | Role |
|--------|------|
| **`x`**, **`y`** | Top-left destination (default **`0`**) |
| **`dirtyX`**, **`dirtyY`**, **`dirtyWidth`**, **`dirtyHeight`** | Sub-rectangle of **`pixelData`** forwarded to **`putImageData`**. If **`dirtyX`** and **`dirtyY`** are both set, the dirty overload is used; **`dirtyWidth`** / **`dirtyHeight`** default to **`pixelData.width`** / **`height`**. |

**`pixelData.width`** / **`height`** are the **`ImageData`** dimensions; placement is **`options.x`**, **`options.y`**.

```typescript
const tile = await painter.pixels.getData(buf, { x: 0, y: 0, width: 32, height: 32 });
// …mutate tile.data in place…
const out = await painter.pixels.setData(buf, tile, { x: 100, y: 40 });
```

## `manipulate(canvasBuffer, options)`

**`PixelManipulationOptions`** (**`lib-next/types/pixels.ts`**):

| Key | Purpose |
|-----|---------|
| **`processor`** | **`(r, g, b, a, x, y) => [r, g, b, a]`** — per-pixel; channels clamped **`0…255`** |
| **`filter`** | **`grayscale`**, **`invert`**, **`sepia`**, **`brightness`**, **`contrast`**, **`saturate`** (implementation uses **`processor`** *or* **`filter`**, not both mixed in one path) |
| **`intensity`** | Scalar for filters (default **`1`**). **`brightness`** uses **`(intensity - 0.5) * 255`** added to RGB |
| **`region`** | **`{ x, y, width, height }`** — only that window is read and written back; rest of image preserved |

```typescript
const warmed = await painter.pixels.manipulate(buf, {
  region: { x: 0, y: 0, width: 200, height: 200 },
  processor: (r, g, b, a) => {
    return [Math.min(255, r + 12), Math.min(255, g + 4), b, a];
  },
});

const gray = await painter.pixels.manipulate(buf, { filter: "grayscale" });
```

## `getColor` / `setColor`

Convenience wrappers around a **1×1** **`getData`** / **`setData`** cycle.

```typescript
const { r, g, b, a } = await painter.pixels.getColor(buf, 42, 42);

const stamped = await painter.pixels.setColor(buf, 42, 42, {
  r: 255,
  g: 0,
  b: 0,
  a: 255,
});
```

---

## Performance

These paths use **CPU loops** in JavaScript for **`processor`** / filters. Prefer **`painter.image`** (Sharp) for heavy full-frame work; use **`painter.pixels`** when you need **`ImageData`** semantics, tight regions, or teaching/debugging.
